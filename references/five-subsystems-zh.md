# 五子系统 —— grill 拷问的依据与 WHY

本文件把《Learn Harness Engineering》12 讲的核心思想，压缩成部署 harness 时要落地的**五个可操作子系统**。grill 每问一个子系统，都引用这里的 WHY 帮用户理解“为什么需要它”。

> 说明：lecture 02 从概念上把 harness 拆成「指令 / 工具 / 环境 / 状态 / 反馈」五件。本文件用的是**可操作的五件产物**视角（指令 / 状态 / 验证 / 范围 / 生命周期），两者是同一套思想的不同切面——前者讲“harness 由什么构成”，后者讲“一个仓库要落盘哪些东西”。

贯穿全部子系统的一句话（lecture 02/03）：**仓库即事实源（the repo IS the spec）。** 模型之外的工程基建，决定了模型能力能被兑现多少。所有上下文都应活在仓库里——结构化指令、显式验证命令、清晰目录，而不是活在对话历史里。

---

## 1. 指令 (Instructions) —— `AGENTS.md` / `CLAUDE.md`

**WHY（lecture 02、04）**
- agent 进项目时和“没有文档、没人带”的新员工一样，只能看到你摆在它面前的文件。指令文件就是它的入职手册：启动路径、工作规则、完成定义。
- 但**别写成一个巨型文件**。lecture 04 的教训：指令膨胀到 600 行后，agent 反而更差——上下文预算被吃光、关键约束“迷失在中间”（Lost in the Middle）、不同时期加的规则互相矛盾。
- 正确做法是**渐进式披露**：根指令文件短小，只放路由和不可变约束（invariants），把项目细节拆进 `docs/` 等专门文档，让 agent 按需读。

**grill 要问出**：用 AGENTS.md 还是 CLAUDE.md（面向 Claude 选后者）；项目用途一句话；启动时除默认 6 步外还要做什么（如指定 Node 版本、先读某架构文档）。

**已有产物时**：若项目已有 AGENTS.md/CLAUDE.md，一句确认“保留并增补，还是重构”，不重复盘问。

---

## 2. 状态 (State) —— `feature_list.json` + `progress.md`

**WHY（lecture 08）**
- agent 不知道你说的“完成”是什么意思。你说“加购物车”，它可能只写个 Cart 组件就声称完成。**feature list 不是给人看的备忘录，而是整个 harness 的地基**：调度器靠它挑任务、验证器靠它判完成、交接报告靠它生成摘要。
- 功能状态必须是**机器可读、外部化**的文件，活在仓库里，不能埋在非结构化的对话文本里。
- 好的进度记录能把下次会话的启动诊断时间砍掉 60-80%（Anthropic 工程数据）。

**grill 要问出（Scope 阶段会复用）**：这个项目接下来要做哪 3-5 件具体功能？每件的依赖关系？每件“完成”的判定标准是什么？——把答案写成真实的 feature 条目，问不出来才退回占位模板。

---

## 3. 验证 (Verification) —— `init.sh`

**WHY（lecture 09、10）**
- 神经网络系统性地**过度自信**（Guo et al. 2017）：agent“感觉”完成了，实际差得远。harness 必须用**外部化、基于执行**的验证，取代 agent 的“感觉”。
- lecture 09 的三层终止检查：先 lint/typecheck → 再跑测试和启动检查 → 最后跑完整用户流程，三层全过才算完成。
- lecture 10：**只有端到端跑通才算真验证。** 单元测试有系统性盲区（接口不匹配、跨层状态、资源生命周期、环境依赖），各部件单独“正确”，一接起来就出问题。而且 agent 一旦知道工作会被 e2e 验证，写代码时的行为也会变好（考虑组件交互、尊重架构边界、处理错误路径）。

**grill 要问出**：装依赖、类型检查、lint、单测、构建分别用什么命令？有没有能跑的端到端/启动检查？——这些依次写进 `init.sh`（`set -e` 快速失败）。scan.mjs 已给出推荐命令，确认或修正即可。

---

## 4. 范围 (Scope) —— 功能依赖 + 完成定义

**WHY（lecture 07、08）**
- agent 容易**既越界又虎头蛇尾**（lecture 07）：要么顺手改一堆无关文件，要么功能只做一半就声称完成。
- 对策是**一次一个功能**：从 feature list 挑且只挑一个未完成功能，做完、验证、记录，再动下一个。配合显式的功能依赖和“完成的定义”，让范围有边界、收尾有闸门。

**grill 要问出**：是否强制“一次一个功能”（默认是）；有没有额外的范围红线（如“不引入新运行时依赖”）；“完成的定义”清单具体包含哪几条。

---

## 5. 生命周期 (Lifecycle) —— `session-handoff.md` + 结束流程

**WHY（lecture 05、06、12）**
- 长任务会**丢失连续性**（lecture 05）：上下文无限累积、跨会话断片。lecture 06：初始化值得有自己独立的一个阶段（先跑 `init.sh` 确认基线健康，再动手）。
- lecture 12：**每次会话都必须留下干净状态。** 下次会话要能立刻跑 `init.sh` 重启、读 progress/handoff 接上上下文，而不是从废墟里考古。
- 这就需要显式的恢复路径和结构化的进度追踪——会话结束前更新 progress.md、feature_list.json，记录阻塞、改动文件、下一步建议。

**grill 要问出**：是否需要 `session-handoff.md`（多会话/较大任务建议要）；会话结束流程除默认步骤外还要补什么。

---

## grill 的“针对性”原则

scan.mjs 会把每个子系统标成 `present` / `partial` / `missing`：
- **missing**：深入 grill，把 WHY 讲清楚，逐项问出决策。
- **partial**：指出薄弱点（如“有 AGENTS.md 但没写完成定义”），针对缺口问。
- **present**：一句话确认“保留 / 增补 / 重构”，不浪费用户时间重复盘问。
